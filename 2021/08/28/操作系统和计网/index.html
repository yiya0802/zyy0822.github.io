操作系统补充：
系统调用：
概述：我们运行的程序基本都是运行在用户态，但要用到关于系统方面的调用，如用户管理，内存管理，进程管理，就要用到系统调用了，由操作系统代替我们完成
具体功能：
设备管理，文件管理，进程控制，进程通信，内存管理

从jvm的角度分析进程与线程的关系
一个进程有多个线程，多个线程共享进程的堆和方法区（元空间），但每个线程都拥有自己的程序计数器，本地方法栈，虚拟机栈
线程是进程划分更小的运行单位，一个进程在运行是可以产生多个线程，进程是独立的，而线程可能是会相互影响的，线程执行开销小，不利于资源的管理和保护

进程的状态：
创建
运行
就绪
阻塞
结束

进程之间通信：
因为进程都有不同的地址空间，任意一个进程的全局变量在另一个进程中都看不见，所有进程之间通信必须通过内核，从内核中开辟缓存区
进程1把数据拷贝到缓存区，进程2再从缓存区把数据读走
1.匿名管道 因为管道是半双工的，所有需要建立两个管道，实质是个循环队列，一个数据只能读写一次，当缓存区满的时候，有一定的规则控制进程进入等待队列
特点：匿名，单向，缓存区有限，只能用户亲缘关系通信
2.有名管道
特点：以有名管道的文件形式存在文件中，先进先出，名字存文件，内容存内存
3.信号
信号可以在任意时候发给另一进程，如果被进程设置为阻塞，则传递被延迟，直到变成运行态才传递给进程
4.消息队列
是一个消息的链表，存在内核中（管道存在文件，内存或磁盘中）只有内核重启或人工删除，消息队列才会被删除
5.共享内存
是多个进程可以直接读写同一块内存，效率高。由于多个进程共享内存，所有需要信号量来控制同步和互斥
6.信号量
用户共享内存时同步进程.用于多线程共享内存的访问
信号量与互斥量的区别
同步是在互斥的基础上，通过其他机制实现访问者对资源的有序访问。互斥量只能是0和1，一个互斥量只能用于一个资源的互斥访问。
信号量可以实现多个资源的多线程互斥和同步。互斥量的加锁解锁必须由同一个线程对应使用，而信号量可以是一个线程加锁，另一个线程解锁
7.套接字
此方法主要用于在客户端和服务器之间通过网络进行通信。套接字是支持 TCP/IP 的网络通信的基本操作单元，可以看做是不同主机之间的进程进行双向通信的端点，简单的说就是通信的两方的一种约定，用套接字中的相关函数来完成通信过程。

计算机网络补充：
应用层：应用进程间的交互通信，DNS HTTP SMTP
传输层：两台主机进程间通信提供通用的数据传输服务，复用：多个应用层进程同时使用下面的传输层服务，分用：传输层的应用信息交给应用层处理
网络层：选择路由和交换节点，确保数据报文的传输
链路层：将网络层的数据报封装成帧
物理层：实现二进制流的传输

tcp udp区别：
tcp面向连接，只能点到点，可靠，全双工，面向字节流，首部长
首先利用三次握手四次挥手来保证信道的可靠性，然后利用ARQ协议（回退N，goback-N，超时重传）保证传输的正确性，滑动窗口来保证传输的及时性，保证流量控制，最后利用慢开始，拥塞避免，快重传，快恢复来进行拥塞控制
udp是无连接，不可靠，面向报文，没有拥塞控制，首部开销小，支持一对一，一对多的交互通信

get与post请求的区别：
GET 用于获取资源，而 POST 用于传输实体主体。
参数不同：get一般是以查询字符串出现在url中，而post一般是存储在实体主体中
安全性：get安全，post不安全

三次握手四次挥手
刚开始客户端处于 closed 的状态，服务端处于 listen 状态。然后
1、第一次握手：客户端给服务端发一个 SYN 报文，并指明客户端的初始化序列号 ISN©。此时客户端处于 SYN_Send 状态。
2、第二次握手：服务器收到客户端的 SYN 报文之后，会以自己的 SYN 报文作为应答，并且也是指定了自己的初始化序列号 ISN（Initial Sequence Number），同时会把客户端的 ISN + 1 作为 ACK 的值，表示自己已经收到了客户端的 SYN，此时服务器处于 SYN_REVD 的状态。
3、第三次握手：客户端收到 SYN 报文之后，会发送一个 ACK 报文，当然，也是一样把服务器的 ISN + 1 作为 ACK 的值，表示已经收到了服务端的 SYN 报文，此时客户端处于 establised 状态。
4、服务器收到 ACK 报文之后，也处于 establised 状态，此时，双方以建立起了链接
作用：确认接收，发送能力是否正常，然后发送初始化序列号，为后来的可靠传输做准备

刚开始双方都处于 establised 状态，假如是客户端先发起关闭请求，则：
1、第一次挥手：客户端发送一个 FIN 报文，报文中会指定一个序列号。此时客户端处于CLOSED_WAIT1状态。
2、第二次握手：服务端收到 FIN 之后，会发送 ACK 报文，且把客户端的序列号值 + 1 作为 ACK 报文的序列号值，表明已经收到客户端的报文了，此时服务端处于 CLOSE_WAIT2状态。
3、第三次挥手：如果服务端也想断开连接了，和客户端的第一次挥手一样，发给 FIN 报文，且指定一个序列号。此时服务端处于 LAST_ACK 的状态。
4、第四次挥手：客户端收到 FIN 之后，一样发送一个 ACK 报文作为应答，且把服务端的序列号值 + 1 作为自己 ACK 报文的序列号值，此时客户端处于 TIME_WAIT 状态。需要过一阵子以确保服务端收到自己的 ACK 报文之后才会进入 CLOSED 状态
5、服务端收到 ACK 报文之后，就处于关闭连接了，处于 CLOSED 状态。

HTTP与HTTPS
开销：HTTPS需要CA证书，SSL加密传输，需要更多的内存，cpu而http是超文本协议传输，明文传输
端口：HTTP是80端口，HTTPS是443端口
安全性：HTTP 的连接很简单，是无状态的；HTTPS 协议是由 TSL+HTTP 协议构建的可进行加密传输、身份认证的网络协议，比 HTTP 协议安全
